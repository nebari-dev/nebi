# Pixi Prerequisite Knowledge

Nebi builds on top of [Pixi](https://pixi.sh). While Nebi's CLI is designed to be approachable, you'll get the most out of it if you understand certain Pixi concepts first.

## What You Should Know

### pixi.toml

This is the manifest file that defines a Pixi workspace. It specifies:

- **Dependencies** — conda and PyPI packages your environment needs
- **Environments** — named configurations (e.g., `default`, `cuda`, `test`)
- **Tasks** — shell commands you can run with `pixi run`
- **Channels** — package repositories to search (e.g., `conda-forge`)
- **Platforms** — which OS/architecture combinations to support

Nebi syncs this file (along with `pixi.lock`) when you push, pull, or publish. If you're not comfortable editing `pixi.toml` and adding dependencies, start with the [Pixi documentation](https://pixi.sh/latest/) first.

### pixi.lock

The lock file pins exact package versions and hashes. It's generated by Pixi when you run `pixi install` or add dependencies. The lock file ensures that everyone who installs from the same `pixi.toml` gets identical packages.

Nebi syncs the lock file alongside `pixi.toml`. This is what makes pulled environments reproducible — the lock file removes solver nondeterminism.

### Environments

Pixi supports multiple named environments within a single workspace. For example, you might have a `default` environment for everyday work and a `cuda` environment with GPU libraries.

Nebi's `shell` and `run` commands can target specific environments by passing Pixi flags:

```bash
nebi shell my-project -e cuda
nebi run my-project test -e test
```

### Tasks

Pixi tasks are named commands defined in `pixi.toml`:

```toml
[tasks]
lab = "jupyter lab"
test = "pytest tests/"
```

You can run them through Nebi:

```bash
nebi run my-project lab
nebi run my-project test
```

## What Nebi Adds

Nebi doesn't replace any Pixi functionality. Here's how the responsibilities divide:

| Pixi handles | Nebi handles |
|---------------|--------------|
| Installing packages | Tracking workspaces by name |
| Solving dependencies | Pushing/pulling specs to a server |
| Managing environments | Versioning and tagging |
| Running tasks | Diffing and status checking |
| Lock file generation | Publishing to OCI registries |

You use Pixi directly for package management and environment setup. You use Nebi for sharing, versioning, and collaborating on those environments.

## Getting Started with Pixi

If you're new to Pixi, these resources will help:

- [Pixi documentation](https://pixi.sh/latest/) — Official docs
- [Pixi getting started](https://pixi.sh/latest/getting_started/) — First steps tutorial
- [pixi.toml reference](https://pixi.sh/latest/reference/pixi_manifest/) — Manifest file format

Once you're comfortable creating a Pixi project, adding dependencies, and running tasks, you're ready to use Nebi.
